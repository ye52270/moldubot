# 향후개선

## 문서 목적
- MolduBot의 현재 구조적 문제/비효율을 정리하고, 다음 개발 사이클에서 바로 실행 가능한 개선 로드맵을 제공한다.
- 본 문서는 "무엇을 왜 고치는지"와 "어떻게 검증할지"를 함께 정의한다.

## 현재 아키텍처 요약
- 실행 축: 단일 deep agent + tool-calling
  - 기준 코드: `app/agents/deep_chat_agent.py`
- API 축: `/search/chat`, `/search/chat/stream`가 에이전트 오케스트레이션 담당
  - 기준 코드: `app/api/routes.py`
- 도구 축: 메일/검색/회의실/일정/ToDo 도구를 단일 레지스트리로 제공
  - 기준 코드: `app/agents/tools.py`
- 미들웨어 축: 의도 주입, 모델 출력 가드, HIL, 후처리, 도구 오류 가드
  - 기준 코드: `app/middleware/registry.py`, `app/middleware/agent_middlewares.py`
- Studio 축: `langgraph dev`용 엔트리 별도 구성
  - 기준 코드: `app/agents/langgraph_entry.py`, `langgraph.json`

## 핵심 문제 진단

### P0-1. 전역 mutable 상태(`current_mail`)로 인한 요청 간 오염 위험
- 증상:
  - `_MAIL_SERVICE` 싱글턴에 `self._current_mail` 저장.
  - 라우트가 `prime_current_mail()/clear_current_mail()`를 수동 제어.
- 위험:
  - 동시 요청에서 사용자 A/B 컨텍스트가 섞일 수 있음.
  - 재현이 어려운 간헐적 품질 저하/오답 유발.
- 근거 파일:
  - `app/agents/tools.py`
  - `app/services/mail_service.py`
  - `app/api/routes.py`

### P0-2. `/search/chat`와 `/search/chat/stream` 오케스트레이션 이중화
- 증상:
  - scope 확인, 메일 컨텍스트 로딩, agent 실행, metadata 구성 로직이 양쪽에 중복.
- 위험:
  - 버그 수정 시 한 경로만 반영될 가능성 큼.
  - REST와 SSE의 동작 불일치 회귀 리스크.
- 근거 파일:
  - `app/api/routes.py`

### P1-1. 대형 파일 집중으로 유지보수성 저하
- 상태:
  - `app/api/routes.py` 640 lines
  - `app/api/bootstrap_routes.py` 928 lines
  - `app/agents/tools.py` 574 lines
- 위험:
  - 책임 경계 불명확, 리뷰/회귀 비용 증가.
  - 변경 충돌 빈도 증가.

### P1-2. Studio 엔트리와 런타임 그래프 초기화 경로 분리
- 증상:
  - 유사한 그래프 초기화 로직이 `deep_chat_agent.py`와 `langgraph_entry.py`에 분산.
- 위험:
  - 프롬프트/툴/미들웨어 변경 시 drift 발생 가능.

### P2-1. 미사용/저사용 서비스 잔존 가능성
- 의심 대상:
  - `app/services/task_execution_service.py`
- 위험:
  - 실제 경로와 무관한 코드가 인지부하를 증가시킴.

## 공식 가이드 기반 개선 방향
- LangGraph/LangChain 공식 문서 기준으로:
  - 병렬 fan-out(`Send`)과 Functional API의 병렬 task(`@task`)는 유효한 개선 수단.
  - 다만 패턴은 상황별 선택이 원칙이며, 현재 MolduBot는 먼저 상태/경로 정리 우선.
- 참조 링크:
  - Router + Send: `https://docs.langchain.com/oss/python/langchain/multi-agent/router`
  - Functional API 병렬: `https://docs.langchain.com/oss/python/langgraph/use-functional-api`
  - 멀티에이전트 개요: `https://docs.langchain.com/oss/python/langchain/multi-agent`

## 권장 로드맵

### Phase 1 (P0): 안정성 우선
1. `current_mail` 전역 상태 제거
- 목표:
  - 요청/스레드 스코프 기반으로 메일 컨텍스트 전달.
- 방법:
  - tool 입력 payload에 `mail_context`를 명시 주입하거나 thread_id 기반 저장소 도입.
  - `prime_current_mail()/clear_current_mail()` 직접 제어 제거.
- 완료 조건:
  - 동시성 테스트에서 요청 간 컨텍스트 누수 0건.

2. chat/stream 공통 오케스트레이터 추출
- 목표:
  - 단일 핵심 함수에서 공통 로직 수행, 출력 포맷만 REST/SSE 분기.
- 방법:
  - 예: `app/api/search_chat_orchestrator.py` 신설.
  - 공통 반환 계약: status/thread/answer/metadata.
- 완료 조건:
  - `/search/chat`, `/search/chat/stream` 회귀 테스트가 동일 계약 기준으로 통과.

### Phase 2 (P1): 구조 정리
1. 라우트 분리
- `bootstrap_routes.py`를 도메인별 모듈로 분리
  - meeting, calendar, legacy_portals, qa, hitl_confirm 등.
2. tool 모듈 분리
- `tools.py`를 기능 축으로 분리
  - mail_tools, search_tools, meeting_tools, calendar_tools, todo_tools.
3. 그래프 팩토리 단일화
- `build_chat_graph(mode=runtime|studio)` 형태로 팩토리 통합.

### Phase 3 (P1~P2): 성능 최적화
1. 병렬화 후보 PoC
- 후보 A: 다건 ToDo 생성
- 후보 B: 다건 일정 생성
- 후보 C: 독립 조회 fan-out 후 결과 통합
2. 적용 방식
- I/O 병렬은 `@task` 또는 async gather 기반으로 소규모부터 적용.
- 큰 아키텍처 전환(Swarm)은 별도 브랜치 PoC 후 채택 여부 결정.
